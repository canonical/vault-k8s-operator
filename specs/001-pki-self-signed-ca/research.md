# Research: PKI Self-Signed CA

**Feature**: PKI Self-Signed CA  
**Date**: 2026-05-25  
**Plan**: [plan.md](plan.md)

## Decision: Vault Native CA Generation

**Chosen approach**: Use Vault's `pki/root/generate/internal` endpoint to generate the self-signed CA directly within Vault.

**Rationale**:
- Aligns with the feature description: "using a Vault self-signed CA"
- The CA private key is generated by Vault's crypto engine, not in Python
- Vault returns the private key in the response, which we store in a Juju secret
- Consistent with how Vault PKI is designed to work

**Alternatives considered**:
- Python `charmlibs.interfaces.tls_certificates.generate_ca()` + `generate_private_key()`: Rejected because it generates the key outside Vault and then imports it, which is less aligned with "Vault self-signed CA"

## Decision: Single Self-Signed CA as Intermediate

**Chosen approach**: Import the self-signed CA directly into the existing `charm-pki` mount as the intermediate CA. No separate root CA mount.

**Rationale**:
- Minimal change to existing PKI architecture
- `PKIManager` already assumes an intermediate CA; this keeps that invariant
- Backward compatible with external CA mode (same code path after import)
- Simpler rotation story (new issuer in same mount)

**Alternatives considered**:
- Root CA + Intermediate CA hierarchy: Rejected as scope creep. The user explicitly confirmed "A sounds better" for the simpler approach.

## Decision: Juju Secret for CA Private Key

**Chosen approach**: Store the self-signed CA private key in a Juju application secret with a fixed label.

**Rationale**:
- Juju secrets are encrypted at rest and access-controlled
- Application secrets are accessible to all units (leader can write, others can read)
- Existing `JujuFacade` has all needed methods (`set_app_secret_content`, `get_secret_content_values`)
- Consistent with how the charm already stores other sensitive data (e.g., autounseal tokens)

**Secret label**: `vault-pki-self-signed-ca`

## Decision: Explicit Mode Branch in PKIManager

**Chosen approach**: Add a `self_signed_ca: bool` parameter to `PKIManager.__init__()` and branch explicitly in `configure()` and `sync()`.

**Rationale**:
- Keeps self-signed and external CA code paths clearly separated
- Easier to test each path independently
- Less risk of accidentally mixing the two modes
- The external CA path remains completely unchanged when `self_signed_ca=False`

## hvac API Research

The `hvac` library's `Client.secrets.pki.generate_root()` method:

```python
client.secrets.pki.generate_root(
    type="internal",
    mount_point="charm-pki",
    common_name="Vault Self-Signed CA",
    ttl="87600h",
    # Optional: country, province, locality, organization, ou
)
```

Response structure:
```json
{
  "data": {
    "certificate": "-----BEGIN CERTIFICATE-----\n...",
    "private_key": "-----BEGIN RSA PRIVATE KEY-----\n...",
    "serial_number": "...",
    "issuing_ca": "..."
  }
}
```

The `certificate` and `private_key` fields are what we need to extract and store.
